# 26kmer.top 域名“他人无法访问/访问过多”排障操作清单

> 场景：站长本人可访问，但部分访客出现“访问过多”“无法访问”提示。

## 一、先做 10 分钟快速止血

1. 在 DNS 里临时只保留 **1 个确认健康的 A 记录**（不要多 IP 轮询）。
2. 将 TTL 调低到 **60~300 秒**，方便快速切换。
3. 在 CDN/WAF 中临时放宽首页 `GET /` 限速（避免误杀）。
4. 让 2~3 位出问题用户重试，确认是否恢复。

---

## 二、DNS 层检查（高优先级）

### 1) 核对记录是否一致
- `www.26kmer.top` 与 `26kmer.top` 是否都指向预期 IP。
- 是否存在“历史遗留 IP”仍在生效。
- 是否同时启用了多条 A 记录但后端配置不一致。

### 2) 核对是否命中异常节点
让失败用户执行：

```bash
nslookup www.26kmer.top
```

收集“失败用户解析到的 IP”，如果集中在某个 IP，说明该节点有问题。

---

## 三、CDN/WAF 限流与误拦截检查

### 常见症状
- 报错文案含“访问过多/请求过于频繁/请稍后再试”。
- 状态码常见 `429` 或 `403`。

### 操作建议
1. 查看安全日志（WAF/CC 防护日志），定位具体规则 ID。
2. 暂时放宽以下规则：
   - 同一 IP 对首页的请求频率阈值；
   - 对移动端 UA 的异常判定；
   - 对国内运营商共享出口 IP 的策略。
3. 给首页静态资源路径（如 `/assets/*`）降低安全阈值，避免连带拦截。

---

## 四、HTTPS/证书检查（多 IP 场景重点）

如果配置了多个 A 记录，必须保证每个节点都完成：
- 站点绑定域名 `www.26kmer.top`；
- 正确证书链（未过期、包含中间证书）；
- SNI 配置正确。

若仅其中一个节点证书异常，会导致“部分人好、部分人坏”。

---

## 五、网络路径/地区性问题验证

让失败用户提供：
1. 访问时间（精确到分钟）；
2. 运营商（移动/联通/电信）与地区；
3. 报错截图；
4. 以下命令结果：

```bash
nslookup www.26kmer.top
ping www.26kmer.top
curl -I https://www.26kmer.top
```

用这些信息和服务器/CDN 日志对齐，快速定位是 DNS、WAF 还是链路问题。

---

## 六、推荐稳定方案（长期）

1. **单源站 + CDN**：先保证一个稳定源站，避免多源配置漂移。
2. **灰度切换**：每次改 DNS 或防护规则前先小范围验证。
3. **统一配置管理**：多节点必须用同一套自动化部署与证书续签。
4. **监控与告警**：增加外部可用性监控（不同地区探测）。

---

## 七、你下一步的执行顺序（可直接照做）

1. DNS 只保留 1 个 A 记录，TTL=300。
2. CDN/WAF 暂时放宽首页限速 30 分钟。
3. 让 3 位失败用户复测并回传 `nslookup + curl -I`。
4. 对照日志，确认是否仍出现 429/403。
5. 如果恢复，再逐步收紧规则，找到不误杀的阈值。
